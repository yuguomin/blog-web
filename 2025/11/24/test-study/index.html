转型测试与学习

Posted on 2025-11-24

前言

本篇文章主要是我计划转岗测试后，对这个行业做的思考和学习。

已经有5年没更新过这个博客了，不可否认的是懈怠，去了宇宙厂以后，技术上的更新也都是写在了内部文档上，也没能输出成自己的内容带走。现在休息了蛮久，也算是满血复活，蓄势待发了，职业规划上也重新做了思考。作为自己的记录也作为面试的目标。

文章主要分为以下几个部分，现状概述、为什么想要转测试、计划转以后我做了些什么。

现状概述

从18年学习前端，加入了第一家公司，做了两年的前端开发，到20年进入宇宙条，前后端+lynx，啥啥都干，满4年后离职。算下来gap了一年半了，本来打算回家结婚休息小半年的，结果也是运气爆棚，老婆怀孕加上儿子诞生，因为家里原因不得不多休息了这么久，即使幸福，也是忐忑。

gap期间，多是处理着家里的事情，但嗷嗷待哺的儿子和社会价值感，不可能让我停歇，感谢家里人的帮助，让我也有时间重新思考了接下来的职业规划。

是的，我计划转行做测试。

为什么要转测试

问

一直以来业界对于测试岗位都有几种说法，互联网开发的鄙视链底端，职业竞争力弱替代性强，工作简单重复等等。这些说法不能说是空穴来风。 本身也是受以下几个角度影响：

  1. 业务流程：在大多数的需求场景下，功能性测试还是主要的工作内容，而这个测试往往放在了产品设计与研发之后，很多时候排期紧张压缩的就是测试时间，背锅的可能也是
  2. 行业门槛：功能性测试给人的感觉就是收集需求流程和边界场景，然后“点点点”，缺啥就找开发要，甚至很多测试代码都没写过
  3. 待遇问题：门槛上的问题，肯定影响了行业地位与收入，甚至前司的一线功能测试外包占比很大

造成这些问题的主要原因，是行业的工作内容决定的，产品和研发更偏向于创新和制造为公司带来实际的收益，测试主要是为了保证产品质量，本身不负责产出。

那么问题来了，我为什么要转测试？

答

我对测试行业的看法

在过去的两家公司，我都遇到过优秀的测试，第一家公司，QA的负责人同时负责项目的整体把控，也有功能测试同事想要学习前端代码。在第二家公司， 优秀的测试开发也很多，在我们比较复杂的开发环境下，提供的可视化测试能力甚至能提高开发的问题排查效率，安全性上的扫描，也帮助避免了 很多的漏洞问题。

所以在我看来，如果只想安安静静的当一个纯粹的功能测试人员，那确实没什么前景。往后走都需要点起新的技能树。

  • 行业技能：深入了解一个业务领域，比如金融、工业、游戏领域等，熟悉行业流程与特性，半个产品。
  • 技术技能：深入自动化、性能、安全测试等领域，具有很强的编码能力，相对研发来说知识面需要更广。
  • 管理技能：随着测试的左移和右扩，测试是很清楚和需要把控项目进度和情况的人，项目管理也是很合适的。

其发展路线也是比较多元的。

自我结合

现在都说互联网行业卷，确实挺卷，但卷的永远不会是金字塔尖的那批人，我在思考我接下来规划的时候，目标是如何来提高我 对市场的竞争力，这是我选择测试的核心思路，围绕这一点做了些总结。

  1. 市场行情考量：测试行业对于编码能力的需求越来越高，测试开发是一个能让我将技术与质量保障相结合的方向。同时无论是技术还是管理都有不错的发展。
  2. 个人时间与效率：成为父亲后，个人的精力确实是有限，所以我更追求高效的工作，更希望能通过 “前置设计、自动化与流程优化” 来系统性解决问题，这和测开的工作模式相契合。
  3. “非科班”：我是一个非理工专业出身，在开发初期是我的短板，但在软技能方面及思维习惯上往往能跳出技术惯性，这对于以“用户视角”测试产品更有优势。

当然，仅仅停留在“想”，不能作为最后决定。

计划转以后我做了些什么

系统化学习

首先，系统化学习这个职业的知识与技能，一是为了更加了解这个方向，二是为了后续从业做准备。 找了一本大学的软件测试教程书，结合技术博客和ai，梳理了相关的知识内容。详情欢迎打开 我的飞书文档

coding

然后结合已有技能，用于实践，开发了几个项目

基于playwright + Docker + GitHub Actions做的web自动化测试
基于k6 + my-json-server做的接口性能测试
基于Jest + Supertest做的自动化接口测试脚本

总结

保持学习。
